# סדר עבודה: טיוטה תחילה (draft-first)

**ברירת מחדל — כל שינוי באתר מתחיל בגרסת טיוטה תחת `app/draft/`, לא בקבצים האמיתיים.**

כשמבקשים שינוי עיצובי / UI / תוכן-בקוד:

1. מיישמים אותו **בסביבת הטיוטה בלבד** — עמוד ניסוי חדש תחת `app/draft/<name>/`, או
   עותק מקומי של הקומפוננטה בתוך `app/draft/`. **לא** עורכים את המקור ב-`components/`,
   `app/` (מחוץ ל-`draft/`) או `data/`.
2. מציגים לאישור, ומחכים.

**קידום לפרודקשן רק בהוראה מפורשת.** רק כשנאמר במפורש משהו כמו **"לדחוף לורסל" /
"להעלות לורסל" / "push to Vercel" / "לפרודקשן"** — מעבירים את השינוי המאושר מהטיוטה
אל הקבצים האמיתיים, שמשם ה-auto-commit דוחף ל-`main` וורסל מדפלוי. בלי אמירה מפורשת
כזו — לא מקדמים, לא דוחפים, ולא נוגעים בקבצים שעולים לורסל.

**מנגנון הקידום:** מסמנים קובץ טיוטה בראשו ב-`// @promote-to: <נתיב-אמיתי>` ומריצים
`npm run promote:draft` — הסקריפט מעתיק אותו למקומו האמיתי ומסיר את שורת הסימון
(`-- --dry-run` לתצוגה מקדימה, `-- --push` לדחיפה מיידית). לחלופין מיישמים את הדלתא
ידנית בקבצים האמיתיים.

הרקע הטכני: `app/draft/` נמצא ב-`.gitignore` ולעולם לא מגיע ל-git/ורסל.
תיעוד מלא: `vault_atlabatla/Meeting Notes/draft-sandbox-environment.md`.
